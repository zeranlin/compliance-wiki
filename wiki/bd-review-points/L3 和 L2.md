可以。要证明 L2 可行，不能只讲概念，必须能模拟一个 L3 业务系统来调用它，并跑通一条工程闭环。

我建议用一句话定义可行性证明：

> 如果一个外部 L3 系统不直接读 BD、不直接拼 prompt、不理解知识库内部结构，只通过标准协议就能完成一次审查作业，那么 L2 就是可行的。

也就是说，L2 的可行性不靠“知识写得多”，而靠“能被外部系统稳定调用”。

**一、先定义角色**
L3 模拟系统：`procure-hegui-demo`

它是一个外部业务系统，只负责：

- 接收采购文件。
- 收集项目元数据。
- 发起审查任务。
- 展示问题工作台。
- 接收人工复核。
- 生成业务报告。

它不负责：

- 找 BD。
- 拼 prompt。
- 判断证据窗口。
- 解释法规。
- 去重归并。
- 维护知识。

这些都交给 L2。

L2 知识供给层负责：

- 发现可用能力。
- 路由主题/BD/lint。
- 解析文档证据。
- 装配裁定包。
- 执行能力。
- 输出结构化判断。
- 提供证据链。
- 接收反馈。

**二、工程交互总流程**
完整流程可以拆成 10 步。

```text
L3 创建任务
  ↓
L3 调 L2：discover 能力发现
  ↓
L3 调 L2：prepare 文档证据编译
  ↓
L3 调 L2：plan 生成审查计划
  ↓
L3 调 L2：run 执行能力
  ↓
L2 内部：证据组织 + 知识装配 + LLM/lint 执行
  ↓
L2 返回：结构化结果 + 证据链 + trace
  ↓
L3 调 L2：merge 结果归并
  ↓
L3 展示问题工作台
  ↓
L3 调 L2：observe 回传人工反馈
```

这就是 L2/L3 的工程级交互。

**三、模拟一次真实调用**
假设 L3 收到一个采购文件：

```json
{
  "task_id": "task-20260428-001",
  "file": "丹巴县造林绿化项目.pdf",
  "project_meta": {
    "采购方式": "公开招标",

    "是否专门面向中小企业": false

```

L3 第一步不跑模型，而是问 L2：

```http
POST /l2/discover
```

请求：

```json
{
  "task_id": "task-20260428-001",
  "project_meta": {
    "采购方式": "公开招标",
    "采购类型": "服务",
```

**六、执行能力**
L3 执行计划：

```http

```

请求：

```json

  "task_id": "task-20260428-001",
  "plan_id": "plan-001"
}
```

L2 内部做这些事：

```text
1. 根据 BD02-005 获取能力定义
2. 从 evidence index 召回候选窗口
3. 补充同对象上下文
4. 查询法律依据和判断链
5. 组装 DecisionPackage
6. 调用本地 LLM 或 lint
7. 校验输出 schema
8. 保存 prompt、证据、结果、trace
9. 返回结构化结果
```

L3 拿到结果后，只做业务展示：

```text
问题 1：净利润率作为评分因素
状态：命中
证据：0632-0644
建议：删除利润率评分，改为与项目履约直接相关的客观能力指标。
```

**十、什么叫 L2 可行**
工程上，L2 可行要满足 8 个验收标准。

```text
1. L3 不直接读取 BD 文件，也能发现能力。
2. L3 不拼 prompt，也能执行审查。
3. L3 不理解知识库目录，也能得到适用主题。
4. L3 不做证据召回，也能得到证据链。
5. L3 不做规则解释，也能得到判断依据。
6. L3 不做重复归并，也能得到问题簇。
7. L3 可以追溯每个结果的 trace。
8. L3 可以把人工反馈回传给 L2。
```

只要这 8 条成立，L2 就不是概念，而是工程上可用的知识供给层。

**十一、当前项目如何证明**
用现有项目可以做一个最小 PoC。

当前已有：

- `wiki/checkpoints/`：能力资产。
- `wiki/checkpoint-domains/themes/`：主题资产。
- `validate_checkpoint_cli.py`：临时 run 接口。
- `validation/cli-runs/`：trace。
- prompt/result/evidence：裁定包雏形。
- theme report：归并雏形。

要补的最小工程件是：

```text
1. capability_manifest.json
2. l2_request.schema.json
3. l2_result.schema.json
4. l2_trace.schema.json
5. l2_feedback.schema.json
6. 一个模拟 L3 的 demo_cli
```

模拟 L3 的 CLI 可以叫：

```bash
python scripts/l3_demo_review.py \
  --review-file xxx.pdf \
  --project-type 服务 \
  --category 造林绿化 \
  --intent compliance_review
```

它内部只调用 L2 标准函数：

```python
capabilities = l2.discover(project_meta)
document = l2.prepare_evidence(review_file)
plan = l2.plan(document, capabilities)
run = l2.run(plan)
clusters = l2.merge(run)
report = l3.render(clusters)
```

如果这个 demo 能跑通，就证明 L2/L3 分工成立。

**十二、最终一句话**
L2 的工程可行性证明，不是证明“BD 能判断风险”，而是证明：

> 一个外部 L3 系统只通过标准协议，就能把业务文件交给 L2，获得能力发现、证据组织、判断执行、结果归并、证据追踪和反馈回流，而无需理解 L2 内部知识结构。

这就是 L2 到 L3 的工程级交互过程。
