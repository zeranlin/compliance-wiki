# NBD F1 精准提升执行账本 20260501

上级依据：[[../NBD可运行知识与CLI运行时协议设计|NBD 可运行知识与 CLI 运行时协议设计]]

关联运行：

- F1 报告：`validation/nbd-runs/20260501-001955-case2-full-nbd-qwen35-f1/evaluation/metrics.md`

## 1. 目标

本账本用于推进 NBD 日常审查链路的精准提升。

目标不是增加临时脚本规则，而是在“可运行知识 + 通用运行时”协议下，提升：

1. Document IR 的证据承载稳定性。
2. CandidateWindow 的召回准确率、完整性和去噪能力。
3. Prompt 对小模型的 SOP 执行约束。
4. F1 评测对真实漏报、误报、名称映射差异的归因能力。

## 3. 执行原则

1. 不在 runtime 中写死具体 NBD ID、具体样本词或 gold 答案。
2. NBD markdown 是源码，NBD IR 是编译后的可运行知识。
3. CLI runtime 只解释 NBD IR、Document IR、CandidateWindow 和 Prompt 协议。
4. 任何新增召回逻辑必须来自 NBD markdown / NBD IR 的通用字段，或来自通用 Document IR 结构能力。
5. 每完成一项任务，必须回填执行记录、验证目录、结果指标和结论。
6. F1 提升不能以明显牺牲精确率为代价。案例二精确率警戒线为 `0.8000`。

## 5. P0：漏报归因诊断器

优先级：P0。

状态：done。

依赖：

- `document-ir.json`、`candidates/`、`items/*/prompt.md`、`items/*/result.json`、`evaluation/metrics.json`。

目标：

对每个 gold 漏报自动判断卡在哪一层，而不是人工猜测。

执行项：

- [x] P0-01 扩展或新增 F1 诊断脚本。
- [x] P0-02 对每个漏报读取 gold 行号、风险原文、审查点名称。
- [x] P0-03 检查 Document IR 同号行是否存在，并计算文本覆盖率。
- [x] P0-04 检查对应 NBD candidates 是否包含该行。
- [x] P0-05 检查对应 prompt 是否包含该行。
- [x] P0-06 检查模型输出是不命中、待复核、命中但名称不匹配，还是没有候选。
- [x] P0-07 输出 `diagnosis.json` 和 `diagnosis.md`。

归因类型：

| 类型 | 含义 | 后续动作 |
|---|---|---|
| `document_ir_missing` | Document IR 同号行缺失或文本覆盖不足 | 修 document_compiler |
| `candidate_missing` | Document IR 有证据，但 candidates 未召到 | 修 NBD recall / recall runner |
| `prompt_missing` | candidates 有证据，但 prompt 未包含 | 修 prompt builder / 窗口截断 |
| `model_conservative` | prompt 有证据，但模型输出待复核或不命中 | 修 NBD SOP / prompt 判定协议 |
| `name_mapping_missing` | 模型命中语义正确，但评测名称不匹配 | 修评测别名层 |
| `postprocess_or_output_issue` | 模型输出有命中但结果结构未被评测读取 | 修 output schema / postprocessor |

验收标准：

1. 案例二当前 30 个漏报均有唯一主归因。
2. 每个归因项必须包含证据链：gold -> Document IR -> CandidateWindow -> Prompt -> ModelResult。
3. 诊断脚本不得写死案例二文件名、具体 NBD ID 分支或 gold 答案。
4. 输出文件写入本次 run 的 `evaluation/` 目录。
