## 1. 阶段结论

NBD 第一阶段完成的是一套从“法源/监管/业务材料/旧 BD 经验”到“小模型可执行标准检查点说明书”的生产闭环。

最终状态：

| 状态 | 数量 |
|---|---:|
| maintained | 153 |
| testing | 0 |
| draft | 0 |
| 总覆盖 | 153 |

本阶段的关键成果不是单纯生成 153 个页面，而是跑通了以下链路：

```text
材料来源
-> 分型规划
-> draft 页面生成
-> 召回预检
-> testing 标准化
-> fixture 三类样本验证
-> 真实长文档 smoke
-> 争议项专项调试
-> maintained 基线
```

## 2. 来源体系

NBD 的来源不是单一表格，而是多源孵化。

| 来源 | 作用 | 进入 NBD 的方式 |
|---|---|---|
| 法律法规 | 提供法定依据、禁止边界、政策要求 | 进入审查依据、命中条件、排除条件和法规桥接规划 |
| 监管口径 | 提供可执行审查动作和地方/专项经验 | 转化为审查目标、风险提示、修改建议和边界例 |
| 业务方新版材料 | 提供被业务认可的新审查点主干 | 作为 NBD 标题、规则、风险提示和全量计划的主来源 |
| 老 BD | 提供历史问题域、误报经验和旧规则经验 | 作为素材参与设计，不在 NBD 中体现历史包袱 |
| 真实范本文档 | 验证未知长文档中的召回、分流和噪声控制 | 进入 smoke、候选窗口统计和 maintained 验收 |
| fixture 样本 | 验证小模型是否能稳定执行单点 SOP | 每个 NBD 至少覆盖 positive / negative / boundary |

其中，`通用（深圳）` 是本阶段 153 个 NBD 的主孵化材料；`wiki/boundaries`、`wiki/checkpoint-domains` 和旧 BD 作为方法论与问题域背景参与，但新版 NBD 与旧版 `checkpoints` 保持隔离。

## 3. 三个状态的定义

### 3.1 draft

`draft` 是材料转写态。

进入 draft 的条件：

- 已从新版材料识别出审查点。
- 已生成基础页面结构。
- 已具备初步审查目标、适用范围、风险提示、修改建议和依据。

draft 的限制：

- 定位词可能仍是标题式关键词。
- 候选召回可能为空或过宽。
- 命中条件可能仍是业务材料原文的泛化表达。
- 缺少 positive / negative / boundary fixture。
- 小模型尚未证明能稳定执行。

draft 不能直接供外部业务系统稳定调用。

### 3.2 testing

`testing` 是可验证态。

从 draft 进入 testing 的条件：

- 页面改造成标准 NBD 结构。
- `定位关键词` 升级为 `定位与召回剖面`。
- 明确章节角色词、对象词簇、行为词簇、后果词簇、模式规则、支持上下文词、降权/排除词。
- 明确候选召回规则、上下文读取规则、基础命中条件、命中条件、排除条件和三分流逻辑。
- CLI 能读取页面并形成候选窗口。
- 通过召回预检或初步小模型 smoke，证明不是纯文本说明。

testing 允许存在：

- 真实样本待人工复核比例较高。
- 某些业务口径尚未完全稳定。
- 部分真实正样本不足。
- 候选窗口需要继续压噪。

testing 的目标是进入验证池，而不是马上证明规则完美。

### 3.3 maintained

`maintained` 是维护基线态。

从 testing 进入 maintained 的最低协议：

- fixture 三类样本可稳定分流：
  - positive：应命中。
  - negative：应不命中。
  - boundary：应待人工复核。
- 真实样本 smoke 无调用失败。
- 真实样本 smoke 无 fallback。
- 候选窗口可控，不能出现大量重复窗口或模板噪声失控。
- 命中、待复核、不命中的结果可解释。
- 对仍需业务判断的复杂场景，能正确进入 `待人工复核`，而不是强行命中。

maintained 不表示“永远正确”，而表示该 NBD 已达到可维护、可调用、可继续积累真实样本的基线。

## 4. 标准协议与方法论

### 4.1 页面协议

每个 NBD 页面必须具备稳定执行区：

```text
审查目标
适用范围
定位与召回剖面
候选召回规则
上下文读取规则
基础命中条件
命中条件
排除条件
判断结果分流
风险提示
修改建议
审查依据
```

弹性迭代区包括：

```text
示例
反例
边界例
易误报场景
地区适用说明
调试备注
小模型可执行性自检
```

### 4.2 定位与召回协议

本阶段最关键的方法论是把“定位关键词”升级为“定位与召回剖面”。

标准定位模型：

```text
词簇层 + 模式层 + 章节角色层 + 组合评分层 + 支持上下文层
```

关键原则：

- 不能只用审查点标题做对象词。
- 不能只靠一个关键词形成候选窗口。
- 必须区分强词、中词、弱词和噪声词。
- 必须识别章节角色，而不是硬匹配标题。
- 必须允许支持上下文补足预算、品目、货物清单、公告、评分表、合同等事实。
- 支持上下文不单独构成风险证据。

### 4.3 三分流协议

小模型必须输出三类结论：

| 结论 | 适用条件 |
|---|---|
| 命中 | 正式条款满足基础命中条件和核心命中条件，且未触发排除条件 |
| 待人工复核 | 相关但上下文不足、项目事实不足、法规边界不足或正式效力不明 |
| 不命中 | 无相关高价值候选、仅模板/格式/目录命中，或支持上下文证明合规 |

这套三分流协议是让小模型可用的核心。真实长文档中的 `待人工复核` 不是失败，而是复杂规则的安全阀。

### 4.4 fixture 协议

每个进入 maintained 的 NBD 应至少有三类 fixture：

| fixture | 目的 | 写法要求 |
|---|---|---|
| positive | 证明小模型能命中风险 | 写成采购文件正式条款，不写抽象描述 |
| negative | 证明小模型能识别合规或无关场景 | 明确合法设置、已补充条件或无正式后果 |
| boundary | 证明小模型能进入待复核 | 相关对象存在，但正式章节、项目事实或法规边界不足 |

本阶段反复证明：抽象 fixture 会导致候选窗口为空或模型误判；高质量 fixture 必须像真实采购文件条款。

### 4.5 真实样本 smoke 协议

真实样本 smoke 不要求所有 NBD 都不命中或命中，而是验证以下指标：

- 调用失败数。
- fallback 数。
- 最大候选窗口数。
- 平均候选窗口数。
- 命中/待复核/不命中的可解释性。
- 是否出现明显模板噪声或候选爆炸。

真实样本 smoke 是从“规则能跑”到“规则能在未知长文档里工作”的关键证据。

## 7. 关键方法论沉淀

### 7.1 先标杆，后全量

首批 12 个不是普通样例，而是生产工艺验证器。只有先证明不同类型 NBD 能被小模型执行，才进入批量生成。

### 7.2 先 testing，不硬升 maintained

Batch 2-5 中多次保留 testing，原因是：

- 跨字段事实不足。
- 表格结构化不足。
- 业务口径复杂。
- 真实正样本不足。
- 模板噪声未压住。

这避免了把“不确定”包装成 maintained。

### 7.3 fixture 必须像采购文件

抽象 fixture 会失败。有效 fixture 必须具备：

- 正式章节语气。
- 审查对象。
- 行为或缺失事实。
- 后果或响应约束。
- 明确边界不足点。

### 7.4 候选窗口是成败源头

候选窗口问题分两类：

- 召回为空：对象词太像标题，未覆盖真实文档表达。
- 召回过宽：弱词、模板、格式、目录、承诺函挤占窗口。

解决方式：

- 强化对象词簇。
- 引入章节角色词。
- 使用组合评分。
- 增加降权/排除词。
- 补支持上下文。

### 7.5 待人工复核是产品能力

复杂规则中，`待人工复核` 不是退步，而是小模型的安全分流机制。

尤其适用于：

- 法规边界不足。
- 项目事实不足。
- 是否同一采购包不明。
- 证书性质不明。
- 表格上下文残缺。
- 正式效力不明。

## 8. 第一阶段结束后的维护口径

153 个 maintained 是第一阶段基线，不是终局。

后续维护重点应从“生成页面”转为：

- 积累真实正样本。
- 建立证书性质库、节能/环保目录、强制标准库等外部事实库。
- 强化预算/最高限价、评分表、采购包、品目事实的结构化抽取。
- 将 NBD 与法规桥接进一步清洗。
- 将 maintained NBD 接入业务系统调用和回归测试。

## 9. 一句话总结

NBD 第一阶段完成的是：用 153 个 `maintained` 审查点验证了一套可复制的生产方法，从法源、监管口径、业务材料和旧 BD 经验出发，经过标准页面协议、召回剖面、三分流 SOP、fixture、真实样本 smoke 和专项调试，最终形成可供小模型调用的标准检查点说明书基线。
