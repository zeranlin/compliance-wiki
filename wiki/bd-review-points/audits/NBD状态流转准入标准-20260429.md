## 1. 定位

本文用于固定 NBD 从 `draft`、`testing` 到 `maintained` 的状态定义、准入条件、禁止升级情形和证据要求。

NBD 的状态不是页面完成度标签，而是小模型审查能力的成熟度标签：

```text
draft
-> testing
-> maintained
```

两道核心流转门槛：

```text
draft -> testing：
从“规则想法 / 材料转写”变成“小模型可执行的标准检查点说明书”。

testing -> maintained：
从“能跑”变成“经过 fixture + 真实样本验证，结果稳定、噪声可控、争议可解释”。
```

## 2. 状态定义

### 2.1 draft：材料转写态

`draft` 表示已经从来源材料中识别出一个 NBD 雏形，但尚未证明小模型可以稳定执行。

draft 应具备：

| 项目 | 要求 |
|---|---|
| 审查主题 | 能说清楚这个 NBD 检查什么风险 |
| 来源依据 | 至少来自法规、监管口径、业务材料、深圳通用表、老 BD 经验、真实范本之一 |
| 初步适用范围 | 初步说明适用品目、采购方式、文件位置或地域口径 |
| 初步风险边界 | 初步说明什么情况可能构成风险，什么情况不构成风险 |
| 初步定位思路 | 有候选章节、对象词、行为词或典型表达的起点 |

draft 的限制：

- 可能仍是业务材料原文转写。
- 可能只有标题式关键词，尚未形成完整 `定位与召回剖面`。
- 可能没有 positive / negative / boundary fixture。
- 可能没有跑过 CLI。
- 不能直接作为外部业务系统稳定调用的标准检查点。

draft 的核心判断是：概念成立，但尚未验证。

### 2.2 testing：可验证态

`testing` 表示 NBD 已经被改造成小模型可执行说明书，可以进入验证池，但尚未证明在 fixture 和真实长文档中稳定。

testing 应具备：

| 项目 | 要求 |
|---|---|
| 页面结构 | 已按 NBD 标准结构组织 |
| 召回剖面 | 已从单纯关键词升级为 `定位与召回剖面` |
| 候选规则 | 已说明候选窗口如何进入、如何降噪、如何读取上下文 |
| 判定协议 | 已明确命中、待人工复核、不命中的分流规则 |
| 证据要求 | 已说明需要哪些正式条款、项目事实或支持上下文 |
| CLI 可读 | `scripts/validate_nbd_cli.py` 能读取并形成候选窗口 |
| 初步样本 | 已有或已规划 positive / negative / boundary fixture |

testing 允许存在：

- 真实正样本不足。
- 候选窗口仍需继续压噪。
- 某些法规或业务口径仍需人工确认。
- 真实样本中存在较多 `待人工复核`。
- 个别场景需要补充支持上下文或排除条件。

testing 的核心判断是：说明书已经可执行，可以被验证和调试。

### 2.3 maintained：维护基线态

`maintained` 表示 NBD 已经通过必要验证，达到可维护、可复现、可继续积累样本的基线。

maintained 应具备：

| 项目 | 要求 |
|---|---|
| fixture smoke | positive、negative、boundary 能稳定分流 |
| 真实样本 smoke | 调用成功，无 fallback，候选窗口可控 |
| 判定可解释 | 命中、待复核、不命中均能对应证据、规则和排除条件 |
| 噪声可控 | 目录、模板声明、格式条款、无关章节不造成持续误触发 |
| 争议处理 | 对证据不足或口径不稳的场景进入 `待人工复核`，不硬判 |
| 维护责任 | 后续修改知道应重跑哪些 fixture 和真实样本 |

maintained 不表示该 NBD 永远正确，也不表示覆盖全部真实风险形态。它表示该 NBD 已达到当前阶段可以交付和持续维护的小模型审查基线。

## 3. draft -> testing 准入标准

一个 NBD 从 `draft` 升级为 `testing`，必须满足以下条件。

### 3.1 标准页面结构完成

页面必须至少包含：

```text
审查目标
适用范围
定位与召回剖面
候选召回规则
上下文读取规则
基础命中条件
命中条件
排除条件
判断结果分流
风险提示
修改建议
审查依据
```

如果缺少上述核心执行区，不得升级为 testing。

### 3.2 召回剖面完成

必须从“定位关键词”升级为“定位与召回剖面”。

召回剖面至少应覆盖：

| 层级 | 内容 |
|---|---|
| 词簇层 | 对象词、行为词、后果词、同义词、近义表达 |
| 模式层 | 常见句式、评分表表达、资格条款表达、合同条款表达 |
| 章节角色层 | 资格条件、评分表、采购需求、公告、合同、投标须知等正式章节角色 |
| 组合评分层 | 强词、中词、弱词、支持词、噪声词之间的组合关系 |
| 支持上下文层 | 预算、品目、采购方式、政策属性、货物清单、合同金额等辅助事实 |

禁止仅凭审查点标题、单个关键词或旧 BD 标题升级为 testing。

### 3.3 判定协议完成

必须明确三分流：

| 结论 | 标准 |
|---|---|
| 命中 | 正式条款满足基础命中条件和核心命中条件，且未触发排除条件 |
| 待人工复核 | 相关但证据不足、正式效力不明、项目事实不足、法规边界不足或需要业务判断 |
| 不命中 | 无相关候选、仅模板/目录/格式命中，或上下文证明合规 |

如果页面只能输出“有风险 / 无风险”，不得升级为 testing。

### 3.4 fixture 思路完成

至少应形成三类样本思路：

| 类型 | 目的 |
|---|---|
| positive | 证明小模型能识别应命中的风险条款 |
| negative | 证明小模型能识别合规、无关或已补充条件的场景 |
| boundary | 证明小模型能在证据不足时进入待人工复核 |

fixture 可以在 testing 阶段继续打磨，但必须已经知道要验证什么。

### 3.5 CLI 初步可执行

必须满足：

- CLI 能读取 NBD 页面。
- 能形成候选窗口或明确说明为何该 NBD 属于全文/跨章节判断。
- 输出结构中能承载三分流结果。

如果 CLI 无法读取，或页面不是机器可消费的标准说明书，不得升级为 testing。

## 4. testing -> maintained 准入标准

一个 NBD 从 `testing` 升级为 `maintained`，必须满足以下条件。

### 4.1 fixture smoke 通过

至少验证三类 fixture：

| 类型 | maintained 要求 |
|---|---|
| positive | 应命中，且命中理由能对应正式条款和规则条件 |
| negative | 应不命中，且不命中理由能对应排除条件或合规事实 |
| boundary | 应待人工复核，或在证据充分时给出合理命中/不命中 |

fixture 必须写成接近真实采购文件的正式条款，不得只写抽象描述。

### 4.2 真实样本 smoke 通过

真实样本 smoke 不要求每个 NBD 都必须发现风险，但必须证明：

- API failure 为 0。
- fallback 为 0。
- 候选窗口数量可控。
- 不出现大量重复候选窗口。
- 不出现目录、模板声明、页眉页脚、格式条款等噪声失控。
- 命中、待复核、不命中的结果可以解释。

### 4.3 候选窗口可控

候选窗口控制是 maintained 的硬门槛。

必须做到：

- 高价值正式条款优先。
- 支持上下文可被读取，但不能单独构成风险证据。
- 章节角色错误时应降权。
- 模板性声明、目录项、表头、重复页眉页脚应降权或排除。
- 块解析导致的重复窗口必须被压缩或去重。

如果候选窗口爆炸，或大量重复窗口会干扰小模型判断，不得升级为 maintained。

### 4.4 判定结果可解释

每个 maintained NBD 的结果应能解释：

```text
为什么召回这个窗口
为什么该窗口是正式证据或只是支持上下文
触发了哪些基础命中条件
触发了哪些核心命中条件
是否触发排除条件
为什么输出命中 / 待人工复核 / 不命中
```

无法解释的偶然正确，不得作为 maintained 依据。

### 4.5 争议不硬升

以下情况不得为了追求数量强行升级为 maintained：

- 法规依据或监管口径不清。
- 业务方需要进一步确认。
- 真实样本中正反例不足。
- 同一条款在不同采购方式、品目或地区下结论可能相反。
- 小模型需要依赖页面外复杂事实才能判断，且支持上下文尚未设计好。
- 误报来源尚未归因。

这些 NBD 应保留在 testing，并进入争议池或专项调试计划。

## 5. 禁止升级情形

### 5.1 禁止 draft 升 testing

出现以下任一情形时，不得从 draft 升级 testing：

- 只有标题和风险描述，没有标准执行区。
- 只有关键词，没有召回剖面。
- 没有命中 / 待人工复核 / 不命中的分流规则。
- 不清楚证据来自哪个正式章节。
- 不清楚哪些场景应排除。
- CLI 不能读取或不能产生可审查输入。
- 审查点本身仍无法与其他 NBD 区分。

### 5.2 禁止 testing 升 maintained

出现以下任一情形时，不得从 testing 升级 maintained：

- fixture positive 无法命中。
- fixture negative 持续误报。
- boundary 不能进入待人工复核，且模型强行下结论。
- 真实样本 smoke 出现 API failure 或 fallback。
- 候选窗口大量重复或爆炸。
- 命中理由不能对应正式条款。
- 结果依赖模型臆测，而不是页面 SOP 和证据。
- 争议点没有记录来源、原因和后续处理方式。

## 6. 状态流转证据包

每次状态升级，应至少留下以下证据。

### 6.1 draft -> testing 证据

| 证据 | 说明 |
|---|---|
| 来源记录 | 说明来自法规、监管口径、业务材料、深圳通用表、老 BD 或真实范本 |
| 页面结构 | 标准执行区已齐备 |
| 召回剖面 | 五层召回模型已落地 |
| 判定协议 | 三分流规则已明确 |
| fixture 计划 | 至少有 positive / negative / boundary 的设计 |
| 初步 CLI 结果 | 能读取、能召回或能进入合理全文判断 |

### 6.2 testing -> maintained 证据

| 证据 | 说明 |
|---|---|
| fixture smoke | 三类 fixture 的期望与实际结果 |
| 真实样本 smoke | 调用、fallback、候选窗口、结论分布 |
| 修复记录 | 调整了哪些召回词、排除词、上下文或判定条件 |
| 争议处理 | 保留 testing 或升级 maintained 的理由 |
| 验收记录 | 对应批次或专项验收页 |

## 7. 验收记录格式

建议每次状态流转记录以下字段：

```text
NBD 编号：
NBD 名称：
原状态：
目标状态：
来源依据：
适用类型：
主要风险：
召回剖面变更：
fixture 结果：
真实样本 smoke 结果：
候选窗口情况：
误报/漏报处理：
争议点：
结论：
后续维护动作：
```

结论只能写：

```text
升级为 testing
保留 draft
升级为 maintained
保留 testing
退回 draft
```

不得使用“基本可以”“应该没问题”“暂时通过”作为状态结论。

## 9. 第一阶段适用结论

第一阶段 153 个 NBD 已按本标准的核心精神完成状态闭环：

```text
draft：完成材料转写和初始页面生成
testing：完成标准结构、召回剖面、三分流和初步验证
maintained：完成 fixture smoke、真实样本 smoke、专项争议处理和最终验收
```

截至 2026-04-29：

| 状态 | 数量 |
|---|---:|
| maintained | 153 |
| testing | 0 |
| draft | 0 |

后续新增或修改 NBD 时，应继续按本文执行状态流转，不得跳过 testing 验证直接进入 maintained。
