# NBD-CLI 三轮验证升级申请清单

## 1. 申请背景

本次使用同一份采购文件完成三轮审查：

1. Qwen + 153 NBD 自动审查。
2. gpt-5.4-mini + 153 NBD 自动审查。
3. Codex 独立主审查。

三轮结果显示：现有日常小模型链路可以识别评分、样品、检测报告、认证范围等典型风险，但对资格条件硬伤、商务合同冲突、章节角色、正反语义和表格结构仍存在明显缺口。

本清单的目标不是新增临时规则，而是按“可运行知识 + 通用运行时”协议进行干净升级：

```text
NBD markdown     = 源码
NBD IR           = 编译后的可运行知识
CLI runtime      = JVM
待审文件          = 输入数据
Document IR      = 编译后的文档结构
CandidateWindow  = 中间证据层
Prompt           = 执行指令包
小模型            = SOP 执行器
业务报告          = 运行结果
```

## 2. 升级原则

### 2.1 必须保持的架构边界

1. 业务知识只能进入 NBD markdown。
2. NBD IR 必须从 NBD markdown 编译得到。
3. CLI runtime 只能解释 NBD IR 和 Document IR，不得新增 NBD 专属业务规则。
4. CandidateWindow 必须可回溯到 NBD IR recall 配置和 Document IR block。
5. Prompt 只能渲染 NBD IR、事实摘要和候选窗口，不得偷塞额外判断逻辑。
6. Postprocessor 只聚合和质检，不重新审查，不覆盖模型 verdict。

### 2.2 本次整改目标

1. 补齐资格条件重大风险的召回和 SOP。
2. 修正小模型“已满足却判命中”的正反语义问题。
3. 强化正式章节、模板章节、通用条款的证据权重。
4. 让评分表、合同条款、技术检测注释形成更稳定的候选窗口。
5. 输出业务可读、可复盘、可调优的审查结果。

## 3. 总体整改分层

| 层级 | 当前问题 | 整改方向 | 禁止做法 |
|---|---|---|---|
| NBD markdown | 部分 NBD 召回配置不够表达资格硬伤、反证条件、章节优先级 | 补充机器召回配置、命中条件、排除条件、待复核边界、易误报场景 | 在 CLI 写死某个 NBD 的关键词 |
| NBD IR | recall/sop/output 对运行时约束不足 | 编译出结构化 recall、sop、output、verdict_policy、evidence_policy | 编译后仍让 runtime 回头读 markdown 自由文本 |
| Document IR | 章节角色和表格结构不足 | 强化 section_role、评分表结构、合同条款结构、资格章节结构 | 用业务风险词在 document compiler 中直接下结论 |
| CandidateWindow | 高价值窗口可能被普通窗口淹没，模板噪声可进入主证据 | 支持 primary/support、recall_quality、formal/template 权重、完整性标记 | 按具体 NBD ID 特判排序 |
| Prompt | 小模型会把合规事实判成风险命中 | 强化命中/不命中/待复核分流、反证优先、模板证据限制 | prompt 模板写入某类 NBD 专属规则 |
| 小模型结果 | 命中说明与 verdict 自相矛盾 | 增加通用质量检查，标记异常，进入复核清单 | 后处理直接改判命中/不命中 |
| 业务报告 | 风险聚合可读性仍需提升 | 按业务问题族聚合，展示统计、证据、NBD、整改建议 | 用报告层新增风险判断 |

## 4. P0 升级申请

### P0-01：资格条件章节强制高价值召回

问题来源：Codex 独立审查发现 `0114-0119` 存在明显资格条件硬伤，但前两轮自动 NBD 未作为核心问题输出。

涉及风险：

1. 上级主管单位同意投标函。
2. 年收入、注册资本、平均盈利。
3. 国有资本持股比例。
4. 经营年限。
5. 国家级优秀企业。

整改层级：

| 层级 | 整改内容 |
|---|---|
| NBD markdown | 在 NBD01 资格公平性类中补充“资格条件章节优先召回”配置；补充上级主管同意函、注册资本、营业收入、盈利、股权结构、所有制、经营年限、荣誉称号等风险对象词 |
| NBD IR | recall.primary_terms、formal_evidence_terms、high_value_combinations 中结构化输出上述词簇；sop.hit_conditions 明确“出现在资格要求/资格性审查中即高风险” |
| Document IR | 将“申请人的资格要求”“资格性审查表”“投标人资格要求”识别为 `qualification_primary` |
| CandidateWindow | 对 `qualification_primary` 中包含资格硬伤词的窗口提升为 primary，recall_quality 至少 partial |
| Prompt | 小模型必须优先阅读资格 primary window；若资格窗口明确出现限制条件，不得因其他章节模板内容降级 |
| 业务报告 | 将资格条件硬伤聚合为业务优先级最高的问题族 |

申请改造的 NBD 范围：

| NBD | 需补充方向 |
|---|---|
| NBD01-001 不得限定供应商所有制形式 | 增加“国有资本持股、国有控股、股权结构”等资格窗口召回和命中条件 |
| NBD01-004 不得将注册资本、资产总额、营业收入、从业人员、利润、纳税额设置为资格条件 | 增加“年收入、注册资本、平均盈利、利润”等变体 |
| NBD01-005 不得将企业股权结构、成立年限、经营年限设置为资格条件 | 增加“经营年限不低于、成立时间、股权结构、持股比例” |
| NBD01-006 不得将隐形限制企业组织形式、规模、经营年限等的资质证照作为资格条件 | 增加“上级主管单位同意函、国家级优秀企业、优秀企业认定” |
| NBD01-008 不得将国务院已明令取消的或国家行政机关非强制的资质、资格作为资格条件 | 增加“国家级优秀企业”等荣誉/称号资格条件边界 |

验收标准：

1. 用本样本重跑时，资格条件 `0114-0119` 必须进入 NBD01 类候选窗口。
2. 候选窗口 source 可回溯到 NBD IR recall 配置。
3. 不允许 runtime 出现 `if NBD01-004` 等特判。

### P0-02：NBD 正反语义硬分流

问题来源：第二轮出现“已明确/不构成风险”却输出“命中”的自相矛盾。

典型样例：

1. 已明确投标报价要求，却命中 NBD01-027。
2. 已明确最高限价，却命中 NBD01-022。
3. 售后服务已在采购需求体现，却命中 NBD02-015。
4. 核心产品已载明，却命中 NBD04-002。
5. 付款 10 个工作日内完成，却命中 NBD06-007。

整改层级：

| 层级 | 整改内容 |
|---|---|
| NBD markdown | 每个缺失型、配置型、合规性确认型 NBD 必须补齐“不命中条件”和“反证窗口” |
| NBD IR | sop.exclusion_conditions 必须结构化输出；verdict_policy.not_hit 必须明确“已明确/已满足/已载明”的分流 |
| Prompt | 固定提示：若判断理由包含“已明确、已满足、不构成风险、符合要求”，不得输出命中 |
| ModelReviewResult | execution_trace.exclusion_checks 记录反证是否成立 |
| Postprocessor | 只做异常标记：verdict=命中且 summary 出现反证词时，标记 `verdict_contradiction`，不自动改判 |
| 业务报告 | 将 `verdict_contradiction` 放入“需复核模型输出质量”区，不作为正式业务风险 |

优先整改 NBD：

| NBD | 问题类型 |
|---|---|
| NBD01-022 明确最高限价 | 已明确则不命中 |
| NBD01-027 采购文件须设定明确的投标报价要求 | 已明确报价方式则不命中 |
| NBD02-015 售后服务评分与采购需求对应 | 需求已体现则不命中 |
| NBD04-002 按规定载明核心产品 | 已载明核心产品则不命中 |
| NBD06-007 收到发票后 10 个工作日内完成资金支付 | 已符合期限则不命中 |

验收标准：

1. 第二轮报告中同类自相矛盾输出不再进入业务风险问题。
2. Postprocessor 不覆盖 verdict，只标记质量异常。
3. prompts 中的反证约束可追溯到 NBD IR sop.exclusion_conditions。

### P0-03：正式证据与模板噪声分层

问题来源：缺失型 NBD 容易被声明函模板、通用条款、投标文件格式误导。

整改层级：

| 层级 | 整改内容 |
|---|---|
| NBD markdown | 缺失型 NBD 增加“正式章节优先”和“模板不可单独命中”规则 |
| NBD IR | recall.noise_terms、priority_policy.template_demote、sop.manual_review_boundaries 结构化输出 |
| Document IR | 将投标文件格式、声明函、承诺函、通用政策附件识别为 `template_support` 或 `policy_support` |
| CandidateWindow | 模板窗口只能为 support；缺失型 NBD 若只有 support 窗口，应输出待复核或不命中 |
| Prompt | 明确“不得把目录、模板、通用条款直接当正式风险证据” |
| 业务报告 | 缺失型问题必须展示“已检查的正式章节” |

优先整改 NBD：

| NBD | 需防误报点 |
|---|---|
| NBD04-003 采购文件中须明确所属行业 | 中小企业声明函模板不能单独证明正式章节缺失 |
| NBD05-007 合理设置本国产品声明函 | 格式模板与正式资格/符合性要求要区分 |
| NBD04-002 按规定载明核心产品 | 采购需求正式载明时不得因模板重复误判 |
| NBD05-004 面向中小企业采购项目，不得设置价格扣除 | 通用价格扣除模板需降权 |

验收标准：

1. CandidateWindow 中每个窗口有 `window_type=primary/support`。
2. 缺失型命中不得只引用 `template_support`。
3. 业务报告能说明“正式章节未见”或“仅模板出现”。

### P0-04：评分表结构解析和权重闭合

问题来源：第一轮和第三轮均识别评分权重异常，但具体合计口径不一致，说明表格层级解析不足。

整改层级：

| 层级 | 整改内容 |
|---|---|
| Document IR | 对评分表保留行列结构，识别评分项、权重、评分准则、一级/二级层级 |
| CandidateWindow | 数值类窗口增加 completeness：是否包含评分项名称、权重列、上下级标题 |
| NBD markdown | NBD02-029、NBD02-030 增加“表格层级不清时输出待复核，不强行算唯一总和” |
| Prompt | 要求小模型列出计算过程和不确定点 |
| 业务报告 | 把“总和不等于 100”和“表格层级不清导致权重不闭合”统一表达为业务问题 |

验收标准：

1. 报告不再只给出 97 或 102 的单一脆弱结论，而能说明表格层级疑点。
2. 数值类问题必须展示计算过程。
3. CandidateWindow 可以回溯评分表完整行。

## 5. P1 升级申请

### P1-01：商务合同冲突类窗口

问题来源：Codex 发现履约保证金冲突、付款周期空白、验收期限空白、检测费用承担不合理。

整改层级：

| 层级 | 整改内容 |
|---|---|
| NBD markdown | NBD06 合同履约类补充付款周期、验收期限、履约保证金冲突、检测费用承担、以实际需求为准等召回配置 |
| Document IR | 将合同条款、商务需求识别为 `contract_primary`、`business_primary` |
| CandidateWindow | 支持跨章节同类条款并列，例如履约保证金一处 20%、一处不需要 |
| Prompt | 要求小模型识别“同一事项前后冲突”和“关键期限空白” |
| 业务报告 | 把合同冲突聚合为独立问题族 |

申请改造 NBD：

| NBD | 方向 |
|---|---|
| NBD06-004 不得收取超过规定比例的履约保证金 | 增加比例/是否收取冲突识别 |
| NBD06-007 采购人应当在收到发票后 10 个工作日内完成资金支付 | 增加空白付款周期和超期付款区分 |
| NBD06-010 必须设定履约验收方案 | 增加验收期限空白、验收主体/程序缺失 |
| NBD07-001 不得要求中标人承担验收产生的检测费用 | 增加“检测合格费用承担”边界 |

### P1-02：技术检测报告类 NBD 分层表达

问题来源：三轮均确认检测报告要求过重，但第二轮扩展较激进。

整改内容：

1. 区分“简单参数要求检测报告”“检测报告数量过多”“委托单位限定”“检测机构限定”“检测标准不清”“检测周期不足”。
2. 每个 NBD 只判断自己的风险，不互相覆盖。
3. 技术参数类 CandidateWindow 必须包含参数、证明材料要求、未提供后果三类要素。

申请改造 NBD：

| NBD | 方向 |
|---|---|
| NBD07-008 | 强化简单参数词簇：纤维含量、织物密度、单位面积质量、厚度等 |
| NBD07-014 | 检测报告数量需基于产品项数量和报告要求计算，证据不足则待复核 |
| NBD07-007 | 委托单位限定为投标人时命中；允许制造商/合法持有人时不命中 |
| NBD07-002 / NBD07-003 / NBD07-006 | 区分检测机构资质、CMA 描述、检测标准设定 |
| NBD07-015 / NBD07-016 | 区分区间响应规则缺失和精细参数锁定产品 |

### P1-03：样品要求冲突与主观评分

问题来源：三轮均识别样品评分主观，Codex 额外发现样品标识冲突。

整改内容：

| 层级 | 整改内容 |
|---|---|

## 6. P2 升级申请

### P2-01：NBD markdown 源码标准补强

所有被本次验证触发或漏报的 NBD，应补齐以下段落：

```text
机器召回配置
候选召回规则
上下文读取规则
基础命中条件
命中条件
排除条件
判断结果分流
易误报场景
小模型可执行性自检
```

重点是三类 NBD：

1. 资格条件类。
2. 缺失/配置类。
3. 数值/表格/冲突类。

### P2-02：NBD IR 编译校验增强

compile-nbd 增加 lint：

| 检查项 | 失败条件 |
|---|---|
| recall 完整性 | primary_terms、formal_evidence_terms、noise_terms 全空 |
| sop 完整性 | hit_conditions 或 exclusion_conditions 缺失 |
| verdict_policy | hit/manual_review/not_hit 任一缺失 |
| evidence_policy | 缺失型 NBD 未声明 primary/support 证据边界 |
| source trace | NBD IR 无 source_path 或段落来源 |

### P2-03：运行后质量检查

Postprocessor 只做质量标记，不改判。

新增质量标记：

| 标记 | 触发条件 | 处理 |
|---|---|---|
| verdict_contradiction | verdict=命中，但 summary 出现“已明确/已满足/不构成风险/符合要求” | 进入模型输出质量复核 |
| template_only_hit | 命中证据只有 template_support | 不进入正式业务风险，转待复核 |
| missing_calculation | 数值类命中无计算过程 | 转待复核或报告提示证据不足 |
| primary_window_missing | NBD 要求 primary 证据但候选只有 support | 转待复核 |
| duplicate_same_evidence | 多个 NBD 以同一证据重复输出 | 报告聚合，不改模型原始结果 |

### P2-04：业务报告升级

业务报告需增加以下内容：

1. 运行统计：NBD 总数、模型结果、失败数、不命中、命中、待人工复核。
2. 模型输出质量统计：自相矛盾、模板单证据、数值无计算、primary 缺失。
3. 问题族聚合：资格条件、评分规则、样品检测、技术参数、商务合同、政府采购政策。
4. 每个业务问题必须包含：证据位置、原文摘录、触发 NBD、风险说明、整改建议、证据质量。
5. 待复核事项不得混入正式风险问题。

## 7. 任务清单

本节是后续执行账本。执行时必须按优先级推进；每完成一个任务，应更新复选框、执行记录、验证目录和结论。

### 7.1 P0：资格重大漏报与误报红线

目标：先解决“业务上最不能漏”和“模型输出最不能错”的问题。

- [x] P0-01 固化 0044-A 回归 fixture。
- [x] P0-02 升级 NBD01 资格条件类 markdown 源码。
- [x] P0-03 补齐资格条件类 NBD IR recall / sop / output 编译字段。
- [x] P0-04 增强 Document IR 对资格章节的 `qualification_primary` 识别。
- [x] P0-05 验证 CandidateWindow 能召回 `0114-0119` 资格硬伤。
- [x] P0-06 为缺失型、配置型 NBD 补齐 `exclusion_conditions` 和 `verdict_policy.not_hit`。

## 10. 申请结论

建议按 Batch A、Batch B、Batch C 执行。本次升级的核心不是增加更多规则，而是把业务知识继续收敛到 NBD markdown 源码，让 NBD IR 真正成为“编译后的可运行知识”，让 CLI runtime 继续保持通用 JVM 的角色。

```text
待审文件 + NBD markdown
  -> Document IR + NBD IR
  -> CandidateWindow
  -> Prompt

  -> 业务报告
```

任何新增风险词、反证词、正式证据边界、窗口扩展策略，都必须进入 NBD markdown，再经 NBD IR 编译后运行。
