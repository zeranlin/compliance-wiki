## 1. 手册定位
本手册用于指导如何把业务方提供的新版审查材料孵化为可由小模型执行的 NBD 标杆。

NBD 标杆不是法规摘录、经验总结或旧 BD 改名。NBD 标杆是一份标准检查点说明书，目标是让小模型在真实采购文件中完成候选召回、条款判断、结论分流、风险提示和修改建议输出。

标杆的意义不在于先写满全部 153 个 NBD，而在于形成一套可复制的生产工艺。只有当首批标杆能稳定指导小模型审查，才进入批量生成和全量扩展。

## 2. 标杆定义
一个 NBD 可称为标杆，应同时满足以下条件：

- 问题足够单一：一个 NBD 只解决一个可执行审查问题。
- 位置足够明确：知道优先读取资格条件、评分因素、采购需求、公告、合同、证明材料或品目专项章节。
- 召回足够稳定：能召回关键条款，且候选窗口不过度重复。
- 判断足够分明：命中条件、排除条件、待复核边界均可逐项执行。
- 输出足够可用：风险提示、修改建议和依据可以直接供业务系统展示或二次处理。
- 验证足够闭环：经过召回预检、小模型 smoke、误报漏报分析和手册化沉淀。

## 3. 核心公式
每个 NBD 应按以下公式生成：

```text
一个 NBD = 一个风险对象 + 一个适用位置 + 一套召回剖面 + 一组三段式判断链 + 一组排除条件 + 一个输出动作 + 一套验证样本
```

其中：

- 风险对象：供应商、业绩、证书、保证金、联合体、分包、检测报告、预算政策等。
- 适用位置：资格条件、评分因素、采购需求、招标公告、合同条款、投标文件格式、货物清单等。
- 召回剖面：章节词、对象词、行为词、后果词、排除词和必要支持上下文。
- 三段式判断链：基础命中条件、实质命中条件、结果分流。
- 排除条件：常见误报和合法例外。
- 输出动作：命中、待人工复核、不命中，以及对应风险提示和修改建议。
- 验证样本：至少包含真阳性、真阴性、常见误报和边界样本。

## 4. 最小粒度原则
NBD 的粒度应以小模型能否稳定执行为准。

### 4.1 应拆分的情况
- 风险对象不同。
- 适用章节不同。
- 命中条件不同。
- 排除条件不同。
- 所需项目元信息不同。
- 法规依据或地区口径不同。
- 输出提示和修改建议不同。

示例：

```text
不得将注册资本设置为资格条件
不得将注册资本设置为评审因素
```

这两类虽然对象相同，但适用章节、法律后果和召回位置不同，应拆成不同 NBD。

### 4.2 可合并的情况
- 只是同义表达不同。
- 风险本质一致。
- 判断链完全一致。
- 输出提示和修改建议一致。
- 只是在不同项目中出现了不同案例。

### 4.3 不宜直接作为标杆的情况
- “证书设置合理性”这类过大问题。
- “内容一致性校验”这类需要多处全文比对的问题。
- “需求是否合理”这类没有可执行条件的问题。
- 依赖外部库但外部库尚未建好的问题。

这类问题应先拆成更小的执行子型，或先定义所需元信息和外部知识库。

## 5. NBD 类型体系
不同 NBD 类型应使用不同 SOP。不要用一套模板硬套所有问题。

| 类型 | 典型 NBD | 核心判断 | 主要风险 |
|---|---|---|---|
| 明确禁止型 | 不得限定所有制、注册地、财务规模 | 正式条款中出现禁止对象和准入/评分后果 | 关键词误报 |
| 评分因素型 | 地域业绩评分、注册资本评分 | 确认是评分项，并存在得分后果 | 把需求描述误判为评分 |
| 配置缺失型 | 公告未载明联合体、未设置核心产品 | 先定位正式章节，再判断缺失 | 只靠全文关键词会不稳定 |
| 数值比例型 | 投标保证金、履约保证金、质保期 | 抽取金额、比例、预算或合同金额 | 数值来源混淆 |
| 关联比较型 | 售后服务评分但需求未体现 | 比较评分项与采购需求 | 上下文不足 |
| 合理性判断型 | 证书设置合理性 | 判断品目、需求、证书性质、分值 | 小模型容易保守待复核 |
| 品目专项型 | 家具检测、医疗资质、物业高处作业 | 先识别品目事实，再判断专项要求 | 货物清单或技术参数召回不足 |

## 6. 标准页面结构
NBD 页面必须区分稳定执行区和弹性迭代区。

### 6.1 稳定执行区
以下章节是小模型和 CLI 的主要读取对象，必须保留：

```text
审查目标
适用范围
定位与召回剖面
候选召回规则
上下文读取规则
基础命中条件
命中条件
排除条件
判断结果分流
风险提示
修改建议
审查依据
```

### 6.2 弹性迭代区
以下章节可按类型增删，但标杆 NBD 应尽量补齐：

```text
示例
反例
易误报场景
地区适用说明
调试备注
小模型可执行性自检
```

### 6.3 页面写法要求
- 每条条件短句表达。
- 不写空泛词，例如“综合判断是否合理”。
- 不把法规长文压在执行链前面。
- 不把业务材料原文直接照抄成命中条件。
- 不把案例写成普遍结论。
- 必须明确什么时候待人工复核。

## 7. 定位与召回剖面设计
NBD 召回不是堆关键词，而是面向未知文件设计定位与召回剖面。

范文案例只能帮助发现典型表达，不能作为固定写法。未来文件可能使用不同章节名、不同表格结构、不同术语和不同顺序，因此 NBD 的定位必须同时使用词簇、模式、章节角色、组合评分和支持上下文。

### 7.1 五层定位模型
每个标杆 NBD 都应按五层定位模型设计：

```text
词簇层 + 模式层 + 章节角色层 + 组合评分层 + 支持上下文层
```

五层含义：

- 词簇层：覆盖同义词、近义词、业务别名、地区叫法。
- 模式层：用正则或结构模式抽取金额、比例、时间、分值、标准号、证书名。
- 章节角色层：识别资格、评分、公告、需求、合同、格式、附件等语义角色。
- 组合评分层：按“对象 + 行为 + 后果 + 章节角色”判断候选价值。
- 支持上下文层：补足预算、品目、公告、采购需求、货物清单、政策等判断事实。

### 7.2 词簇层
定位词不能只是固定关键词，应设计成词簇。

每个词簇应区分：

- 强词：单独出现就有较强指向性。
- 中词：需要与章节、行为、后果或模式组合。
- 弱词：只能辅助判断，不应单独召回高价值候选。
- 噪声词：出现后应降权、排除或转入待复核。

示例：`NBD06-003 不得收取超过规定比例的投标保证金`

```text
强词：投标保证金、响应保证金、投标保函
中词：保证金、担保、保函
弱词：金额、比例、提交、退还
噪声词：履约保证金、质量保证金、农民工工资保证金、合同保证金
```

示例：`NBD04-001 招标公告需载明是否接受联合体`

```text
强词：本项目接受联合体投标、本项目不接受联合体投标、是否接受联合体
中词：联合体、联合投标、共同投标、联合响应、联合体协议
弱词：牵头单位、成员单位、协议书
噪声词：联合体协议书格式、投标文件格式、附件、目录
```

### 7.3 模式层
模式层用于识别文本形态，不直接给出风险结论。

常见模式：

```text
金额：\d+(\.\d+)?\s*(元|万元|人民币)
比例：\d+(\.\d+)?\s*%
年限：近[一二三四五六七八九十\d]+年、成立满\d+年
分值：得\d+(\.\d+)?分、不得分、最高得\d+(\.\d+)?分
门槛：不低于、不少于、高于、低于、以上、以下
标准号：GB、GB/T、ISO、ASTM、BIFMA、YY、WS 等标准代号
证书名：.*证书、.*认证、.*资质、.*许可证
```

模式层只能说明“这里有金额、比例、年限、分值、标准或证书形态”。是否构成风险，必须再判断对象、章节角色和后果。

### 7.4 章节角色层
章节角色层用于识别文本功能，而不是硬匹配章节标题。

资格角色可能写成：

```text
申请人的资格要求
投标人资格要求
供应商资格
响应人资格
合格投标人
参与条件
准入要求
资格性审查
资格审查表
```

评分角色可能写成：

```text
评审因素
评审标准
评分细则
综合评分表
商务部分
技术部分
价格部分
评分准则
评审办法
```

公告角色可能写成：

```text
招标公告
采购公告
项目概况
项目基本情况
申请人的资格要求
获取招标文件
提交投标文件截止时间
本项目是否接受联合体
```

采购需求角色可能写成：

```text
采购需求
用户需求书
技术要求
服务要求
商务要求
货物清单
设备清单
技术参数
配置清单
```

合同或格式角色应特别降权：

```text
合同条款
合同格式
投标文件格式
声明函
承诺函
协议书格式
附件
目录
示范文本
```

章节角色应结合标题、邻近结构、表头和后果词综合判断。例如同时出现“项目概况、预算金额、最高限价、采购需求、合同履行期限、本项目是否接受联合体”，即使标题不是“招标公告”，也应识别为公告或项目基本情况区域。

### 7.5 组合评分层
候选价值由组合决定，不能由单个词决定。

高价值候选：

```text
章节角色 + 对象词簇 + 行为词簇
对象词簇 + 行为词簇 + 后果词簇
评分角色 + 对象词簇 + 得分结构
对象词簇 + 数值模式 + 后果词簇
资格角色 + 对象词簇 + 门槛模式
公告角色 + 对象词簇 + 明确结论
```

中价值候选：

```text
对象词簇 + 行为词簇
对象词簇 + 数值模式
章节角色 + 对象词簇
对象词簇 + 后果词簇
```

低价值候选：

```text
只有对象词
只有章节词
只出现在目录、附件、格式、模板、承诺函或示范文本中
只有政策引用，没有项目约束动作
```

低价值候选不应直接挤占主候选窗口，可作为辅助上下文或备用窗口。

### 7.6 支持上下文层
复杂 NBD 必须区分主证据召回和支持事实召回。

主证据召回用于找可能构成风险的正式约束条款。

支持事实召回用于补足判断条件，例如：

- 预算金额、最高限价、合同金额。
- 采购品目：货物、服务、工程。
- 是否专门面向中小企业。
- 招标公告正式段落。
- 采购需求、技术参数、货物清单、设备清单。
- 评分项对应的采购需求依据。
- 是否涉及射线装置、医疗器械、家具检测、物业高处作业等品目事实。

支持上下文不能单独作为命中证据。小模型最终命中仍必须回到：

```text
正式约束条款 + NBD 命中条件 + 未命中排除条件
```

### 7.7 候选窗口控制
标杆 NBD 的召回应追求“找得到、少重复、够判断”。

建议控制口径：

- 简单禁止型：平均候选窗口 3 到 6。
- 评分因素型：平均候选窗口 4 到 8。
- 数值比例型：主候选 3 到 6，支持上下文另行补充预算或合同金额。
- 配置缺失型：不以候选数为唯一指标，必须确认正式章节是否被定位。
- 品目专项型：允许候选稍宽，但必须通过品目事实和排除词降噪。

候选窗口过多时，优先检查：

- 是否把弱词当成强词。
- 是否缺少噪声词降权。
- 是否没有区分主证据和支持上下文。
- 是否只按全文关键词召回，未识别章节角色。
- 是否缺少窗口去重或相邻窗口合并。

候选窗口过少时，优先检查：

- 同义词簇是否太窄。
- 未知章节标题是否未覆盖。
- 是否缺少模式层，如金额、比例、分值、证书名。
- 是否过早把中价值候选过滤掉。

### 7.8 NBD 页面写法
新版 NBD 页面直接使用 `定位与召回剖面` 标题，不再使用 `定位关键词`。

标准结构如下：

```text
章节角色词
对象词簇
行为词簇
后果词簇
数值/模式规则
支持上下文词
降权/排除词
高价值组合
中价值组合
低价值组合
```

验证阶段不保留历史负担。已有样例页如继续参与标杆验证，应重新生成或整体改写为上述干净结构。

## 8. 三段式判断链
每个 NBD 都必须写成三段式判断链。

### 8.1 基础命中条件
基础命中条件回答“这个候选有没有进入判断资格”。

推荐格式：

```text
A：候选条款位于本 NBD 适用章节或正式约束文本。
B：候选条款涉及本 NBD 的审查对象。
C：候选条款可能影响资格、评分、响应、合同、政策适用或文件完整性。
```

### 8.2 实质命中条件
实质命中条件回答“为什么构成风险”。

推荐写法：

```text
同时满足以下条件时命中：
- 条款明确要求或排除某类对象。
- 条款产生准入、得分、扣分、响应无效、合同义务或政策后果。
- 未命中排除条件。
```

### 8.3 判断结果分流
必须把三类结果写清楚：

- 命中：证据足够，且能直接支撑风险提示。
- 待人工复核：候选相关但上下文不足、项目事实不足或法规边界不明。
- 不命中：未召回相关候选，或候选属于排除条件。

待人工复核不是失败。合理的待复核能防止小模型强行下结论。

## 9. 排除条件设计
排除条件是 NBD 标杆的核心质量区。

每个 NBD 至少应覆盖以下误报来源：

- 信息填报项：只要求填写信息，不产生准入或评分后果。
- 投标文件格式：只是响应格式、承诺函、声明函。
- 示范文本或模板：不是本项目正式约束。
- 合同通用条款：只约定争议解决、履约管理或责任分配。
- 政策说明：只是引用法律政策，不直接约束供应商。
- 合法必要条件：法律法规、强制标准或项目属性明确要求。
- 证据不足：不能确认正式效力或项目事实时，进入待人工复核。

标杆 NBD 必须写“反例”，不能只写正例。

## 10. 支持上下文标准
很多 NBD 不能只靠候选窗口完成判断，必须设计支持上下文。

### 10.1 需要支持上下文的类型
- 配置缺失型：需要定位正式公告、项目基本情况或投标人资格要求。
- 数值比例型：需要预算金额、合同金额、保证金金额。
- 关联比较型：需要同时读取评分项和采购需求。
- 品目专项型：需要货物清单、技术参数、设备清单。
- 政策判断型：需要采购品目、预算金额、是否专门面向中小企业。

### 10.2 支持上下文的使用边界
支持上下文只用于补足项目事实，不应单独构成命中证据。

小模型最终命中仍必须回到：

```text
正式约束条款 + NBD 命中条件 + 未命中排除条件
```

## 11. 输出标准
NBD 输出必须让业务系统可直接使用。

### 11.1 结论
外层结论只允许：

```text
命中 / 待人工复核 / 不命中
```

### 11.2 输出性质
输出性质应与 NBD 元数据一致：

```text
风险 / 提醒 / 配置缺失 / 口径错误 / 编制建议 / 其他
```

### 11.3 风险提示
风险提示应说明：

- 文件哪里存在问题。
- 可能造成什么合规风险。
- 不夸大为确定违法，除非法规依据和证据均充分。

### 11.4 修改建议
修改建议应可操作：

- 删除不合理限制。
- 改为与采购需求相关的客观条件。
- 明确是否接受或不接受。
- 补充预算、比例、期限、范围、标准。
- 调整为符合法规或政策的表述。

## 12. 法规映射标准
NBD 必须能说明为什么可以查，但法规映射不应压过执行链。

### 12.1 法规状态
建议使用：

```text
待清洗 / 待核验 / 已桥接 / 已维护
```

### 12.2 地区边界
必须区分：

- national：全国法规或全国性规范可支撑。
- regional：地方政策、地方交易规则或地方业务口径支撑。
- mixed：全国原则加地方细化。

地方口径不得包装成全国通用规则。深圳表是孵化来源，不等于 NBD 最终只服务深圳。

## 13. 验证流程
NBD 标杆必须经过四轮验证。

### 13.1 人工结构审查
检查页面是否完整：

- 是否单一问题。
- 是否写清适用章节。
- 是否有召回剖面。
- 是否有基础命中、命中条件、排除条件。
- 是否有风险提示和修改建议。
- 是否有法规依据和地区边界。

### 13.2 召回预检
使用 `validate_nbd_cli.py preflight` 检查候选召回：

- 是否能召回关键条款。
- 候选窗口是否过多。
- 是否大量重复。
- 是否缺少支持上下文。

召回预检不调用模型，只判断“找不找得到”。

### 13.3 小模型 smoke
使用 `validate_nbd_cli.py run` 调用小模型：

- 至少跑 3 到 5 个不同品目样本。
- 至少覆盖真阳性、真阴性和常见误报。
- 观察命中、待复核、不命中的分布。
- 抽查 `prompt.md` 和单项报告。

### 13.4 误报漏报复盘
复盘时不要只改 prompt，应判断问题来源：

- NBD 粒度过大。
- 关键词召回太宽或太窄。
- 排除条件不足。
- 支持上下文不足。
- 法规边界不清。
- 样本自身需要人工判断。

## 14. 达标门槛
NBD 状态升级应有明确门槛。

### 14.1 draft
满足：

- 页面已生成。
- 来源材料可追溯。
- 基本章节完整。

### 14.2 testing
满足：

- 人工结构审查通过。
- 召回预检能召回关键条款。
- 小模型 smoke 无格式失败。
- 至少能解释主要不命中和待复核原因。

### 14.3 maintained
满足：

- 至少通过一轮多样本验证。
- 真阳性可以命中。
- 常见真阴性可以排除。
- 待复核边界合理。
- 法规依据和地区边界完成清洗。
- 风险提示和修改建议可直接复用。

## 16. 批量生成原则
只有当首批标杆的类型方法稳定后，才进入批量生成。

批量生成时应按类型分批，而不是按 Excel 行号机械推进：

1. 先生成明确禁止型和评分因素型。
2. 再生成数值比例型和履约配置型。
3. 再生成配置缺失型和政策判断型。
4. 最后生成合理性判断型和品目专项型。

每批生成后必须跑召回预检和抽样 smoke，不允许只生成页面不验证。

### 16.1 合理性判断型生成要求
合理性判断型 NBD 不允许只写“与采购需求无关则命中”。生成时必须补齐一套可执行的关联度方法。

最低结构：

- 逐对象拆分：多个证书、资质、荣誉、认证、业绩或设备要求必须逐项判断。
- 对象性质：行政许可、强制认证、执业资格、岗位能力、产品或服务认证、管理体系认证、荣誉称号、企业身份、协会评价。
- 需求锚点：采购标的、服务内容、技术参数、岗位人员、作业场景、货物清单、依法准入要求。
- 关联强度：直接对应、表面相关但必要性不明、明显无关。
- 后果强度：资格准入、响应无效、高分值评分、普通低分值评分。
- 结论分流：明显无关且后果强时命中；表面相关但必要性或分值合理性不明时待人工复核；依法必要或直接对应需求时不命中。

合理性判断型的正例和反例必须成对提供：

- 正例应展示“明显无关 + 准入或高分值后果”。
- 反例应展示“依法必要或直接对应需求”。
- 边界例应展示“表面相关但必要性不明”，并要求输出待人工复核。

## 17. 手册自检清单
使用本手册生成 NBD 前，应逐项确认：

- 这个 NBD 是否只解决一个问题。
- 这个 NBD 属于哪种审查类型。
- 它需要哪些正式章节。
- 它需要哪些支持上下文。
- 它的命中条件是否可逐项打勾。
- 它的排除条件是否覆盖常见误报。
- 它的待复核边界是否明确。
- 它的风险提示和修改建议是否可直接展示。
- 它的地区边界是否明确。
- 它的验证样本是否足够。

## 19. 2026-04-30 IR 化后标杆生成规则

P0-P4 后，NBD 标杆不再以“整页说明书进入 prompt”为目标，而是以“源码可编译、IR 可运行”为目标。

### 19.1 标杆生成的新交付物

一个新 NBD 达到标杆要求时，必须能编译出：

| 产物 | 用途 | 必备内容 |
|---|---|---|
| Recall IR | 找候选窗口 | 主召回词、正式证据词、噪声词、冲突词、完整性要素、候选召回规则、上下文读取规则 |
| Prompt IR | 小模型执行 SOP | 审查目标、适用范围、专项判断方法、命中条件、排除条件、待复核边界、结果分流、风险提示、修改建议、审查依据 |
| Governance IR | 人工治理追溯 | 来源、版本、验证记录、变更记录、审计链接 |

### 19.2 可以进入 Prompt IR 的内容

```text
审查目标
适用范围 / 不适用范围
专项判断方法
基础命中条件
命中条件
排除条件
待人工复核边界
判断结果分流
风险提示
修改建议
审查依据
```

### 19.3 不得进入 Prompt IR 的内容

```text
定位与召回剖面全文
机器召回配置全文
调试备注
小模型可执行性自检
待 smoke / 待预检 / 待验证
空示例 / 空反例占位
验证日志
模型对账结论
```

### 19.4 maintained 页面洁净门槛

新增或维护 NBD 时，页面达到 `maintained` 前必须满足：

1. 无历史占位：不得出现 `待 smoke`、`待预检`、`待验证`、`待通过 fixture`。
2. 无空样例：没有真实内容时删除 `示例`、`反例`，不得写“待补充”。
3. 无治理段落：`调试备注`、`小模型可执行性自检` 进入 audit，不进入 NBD 页面主体。
4. 运行知识完整：边界例、易误报场景、排除条件、待复核边界不得因清理治理记录被删除。
5. 全量编译通过：`compile-nbd` lint error 必须为 0。

### 19.5 标杆验收的新基线

标杆验收不只看页面是否写得完整，还要看运行链路：

```text
NBD markdown
-> compile-nbd 输出三类 IR
-> recall 生成 CandidateWindow
-> build-prompt 只包含 Prompt IR 可执行 SOP
-> run-model 返回三分流 JSON
-> report 形成业务可读报告
```
