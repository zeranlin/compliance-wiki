## 目标
- 为 `wiki/findings/` 建立统一、严肃、可追溯的标准命名和来源分层规则。
- 避免 `finding` 名称直接沿用项目页、扫描页或个人整理材料中的随意标题。
- 确保每个标准风险点都具备来源支撑，且能够向外部业务系统稳定输出。

## 基本原则
- `findings` 是标准规则层，不是项目摘要层，也不是个人经验条目层。
- 标准风险点名称必须先有“来源分层判断”，再决定是否可以进入 `wiki/findings/`。
- 不是所有本地上传文件都能直接作为权威来源，但可以作为重要辅助来源。
- 任何标准 `finding` 都不能只靠单一项目或单一地方人员整理文件直接生成。

## 来源分层
### A 类：法源 / 官方规范层
- 包括：
  - 法律、行政法规、部门规章、规范性文件
  - 财政部正式通知
  - 中国政府采购网正式发布的官方案例、监管文章、专项整治口径
- 用途：
  - 决定风险点的上位问题类型
  - 决定风险点的法律依据
  - 决定风险点名称的权威锚点
- 权威级别：最高

### B 类：官方监管 / 检查口径层
- 包括：
  - 财政部门专项检查清单
  - 官方检查要点
  - 官方问题清单、典型问题汇编
- 用途：
  - 决定“监管实际怎么查”
  - 帮助把法源问题转化为可检查的问题类型
- 权威级别：高

### C 类：地方业务整理 / 内部经验层
- 包括：
  - 地方业务人员整理的 Excel、Word、检查表、经验库
  - 内部训练用检查点
  - 辅助小模型用检查链、思维链、算法规则清单
- 用途：
  - 提供高质量问题拆解
  - 提供地方实践中的高频表达、检查点、SOP 雏形
  - 辅助发现标准风险点候选
- 限制：
  - 不能单独作为权威命名来源
  - 不能单独决定新增标准 finding
  - 必须回到 A / B 类来源做校正
- 权威级别：中

### D 类：项目证据层
- 包括：
  - `raw/full-risk-scans/`
  - `raw/numbered-text/`
  - 源文件批注
- 用途：
  - 证明该风险在真实项目中如何出现
  - 为 `finding` 提供正例、反例和证据边界
- 权威级别：证据层，不是命名权威层

## finding 标准命名规则
### 1. 优先采用“问题类型名”
- 优先写成可核查的问题类型，不写成口号式结论。
- 例如：
  - `法定资格条件进入评分`
  - `评分因素未细化量化`
  - `体系认证直接计分`
- 不推荐：
  - `评分不合理`
  - `明显违法`
  - `存在问题`

### 2. 优先采用“官方上位类 + 业务子类”
- 若官方已有稳定上位问题类，应优先锚定上位类，再决定业务子类名称。
- 例如：
  - 上位类：`设置差别歧视条款`
  - 子类：`本地服务机构或设点承诺直接得分`

### 3. 名称必须支持检索和 SOP
- 小模型看到 `finding` 名称后，应能大致判断需要去原文找什么条款。
- 名称不应过于法学抽象，也不应过于项目化。

### 4. 不得直接沿用项目扫描页自然语言标题
- `raw/full-risk-scans` 中的项目风险标题可以很细，但不能直接原样升格为标准 finding。
- 升格前必须先做：
  - 来源校正
  - 上位类对齐
  - 边界收敛
  - SOP 可复现性检查

## 本地上传材料的使用规则
### 1. 本地上传材料必须先分类
- 每份上传材料进入知识库前，先判断其来源属性：
  - 官方原文副本
  - 地方官方汇编
  - 地方业务整理
  - 内部经验清单
  - 项目辅助材料

### 2. 本地上传材料不能直接等于权威规则
- 地方人员整理的 Excel、检查表，即使质量很高，也只能先归入 C 类来源。
- 只有与 A / B 类来源完成对齐后，才可以反向影响 `finding` 命名和定义。

### 3. 本地上传材料的高价值用途
- 帮助拆解检查点
- 帮助完善 `SOP 3-7`
- 帮助发现现有 finding 的缺口
- 帮助形成地方专项检查子类

## 当前两个本地附件的建议定位
### 1. `四川2023年关于政府采购文件典型问题预计处理0828.xlsx`
- 当前判断：`B/C 之间，偏 C`
- 原因：
  - 内容呈现为“政府采购文件典型问题清单”
  - 含法律依据、是否限制、系统改造方式等列
  - 更像地方问题清单或业务归纳成果
- 建议用途：
  - 作为“地方典型问题目录”的辅助来源
  - 用于补充风险点主目录候选
  - 不直接作为唯一命名依据

### 2. `采购文件歧视条款检查点.xlsx`
- 当前判断：`C 类`
- 原因：
  - 已经高度结构化为检查点、检查事项、校验规则、思维链
  - 明显适合作为小模型或系统规则输入
  - 但本质仍是检查规则设计，不是权威法源
- 建议用途：
  - 作为 SOP 拆解和小模型执行逻辑的重要辅助来源
  - 用于补强 `审查逻辑 / SOP`
  - 不直接单独决定标准 risk finding 名称

## 入库建议
- A / B 类来源建议进入 `law-wiki/` 或与 `law-wiki` 相连的本地规范层。
- C 类来源建议单独设为“地方 / 业务整理材料层”，与法源层区分存放。
- D 类来源继续保留在 `raw/`。

## 判定顺序
未来每次新增或重命名 `finding`，按以下顺序判断：
1. 先映射 [[rules/A类通用检查点目录|A类通用检查点目录]] 中的上位检查点
2. 再找 A 类法源与 B 类官方检查口径
3. 再看 C 类地方业务整理材料如何拆解
4. 最后用 D 类项目证据验证是否真实高频出现

## 结论
- 标准 finding 的名称不能只从 `law-wiki` 直接生成。
- 也不能只从项目扫描页或地方 Excel 自由长出来。
- 正确方法是：
  - A / B 类来源定上位类与权威口径
  - C 类来源补检查点和 SOP
  - D 类来源补项目证据和边界
