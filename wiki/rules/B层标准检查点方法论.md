## 目标
- 定义 B 层标准检查点产品库的定位、字段、状态和使用方式。
- 明确 B 与 A 边界之间的关系。
- 为业务系统读取、组合、启用标准检查点提供稳定口径。

## 产品定位
B 层标准检查点产品库，是面向政府采购招标文件合规审查的标准检查点库。

它回答一个核心问题：

```text
政府采购招标文件具体应该检查哪些问题？
```

B 层不是法规原文库，也不是项目案例库，而是把法源边界、监管口径和业务审查经验加工成可被业务系统读取、组合、启用的标准检查点执行说明书。

每个 B 页面应当同时回答：

```text
查什么？
去哪里查？
看到什么先召回？
满足什么条件才命中？
哪些情况必须排除？
命中后如何表达审查结论？
```

因此，B 不只是检查点名称清单，而是用于支撑业务系统、小模型和人工审查人员稳定执行的审查说明书。

## 产品目标
| 目标 | 说明 |
|---|---|
| 全面 | 尽量覆盖政府采购招标文件合规审查中的主要风险检查点 |
| 权威 | 每个检查点必须能回到至少一个 A 类法源边界或官方监管口径 |
| 稳定 | 检查点名称、编号、字段结构保持稳定 |
| 可读 | 业务人员、小模型、系统都能理解检查点含义 |
| 可组合 | 外部业务系统可按任务和场景自行选择检查点 |
| 可治理 | 每个检查点有状态、成熟度和审查模式，便于持续维护 |

## 独立性原则
B 是全新的、独立的标准检查点产品线。

B 的设计只接受通用化后的审查能力、判断逻辑和表达结构，不继承历史实验页面的身份、名称体系、链接关系或兼容关系。

B 页面中不得出现历史实验页面来源、旧名称、迁移说明、兼容说明或外部依赖关系。凡进入 B 的内容，都必须被重写为标准检查点语言、审查执行语言和法源边界语言。

## 吸收口径
B 可以参考既有审查材料中成熟的内容格式，但吸收的是能力，不吸收来源身份。

可吸收内容包括：
- 问题定义方式。
- 审查逻辑结构。
- 定位关键词组织方式。
- 候选条款召回方法。
- 命中条件和排除条件。
- 正例和反例表达方式。
- 结论表达口径。
- 法源边界和监管口径组织方式。

不得吸收内容包括：
- 历史页面名称。
- 历史页面链接。
- 历史兼容关系。
- 单项目样本结论。
- 未经重写的风险定性表述。
- 脱离 A 边界的经验判断。

## 与 A 的关系
A 是法源边界，B 是标准检查点。

```text
A 说明为什么能查。
B 说明具体查什么。
```

每个 B 必须挂接至少一个 A。A 不直接作为检查点使用，B 是实际审查任务中可选择、可调用的检查点。

## BD 字段
| 字段 | 是否必填 | 说明 |
|---|---|---|
| BD编号 | 必填 | 问题域内唯一编号，格式为 `BDxx-yyy`，如 `BD01-001` |
| BD 问题域 | 必填 | 所属问题域，如 `BD01` |
| 标准检查点 | 必填 | 检查点名称 |
| 对应A边界 | 必填 | 至少一个 A-BD |
| 审查问题句 | 必填 | 用一句话说明检查什么，建议以“是否……”开头 |
| 权威依据 | 必填 | 主要法源或官方监管口径 |
| 审查模式 | 必填 | 说明适合何种审查数据条件 |
| 成熟度 | 必填 | 高 / 中 / 低 |
| 状态 | 必填 | 已设计 / 候选 / 可启用 / 非静态 / 已合并 / 已废止 |
| 备注 | 选填 | 适用边界、待补事项、特殊说明 |

## 审查模式
| 审查模式 | 含义 |
|---|---|
| 静态文件 | 只看采购文件 / 招标文件即可判断 |
| 需上下文 | 需要采购文件 + 项目品类、金额、地区、政策背景 |
| 交易数据 | 需要投标文件、报价、保证金、交易日志等 |
| 履约数据 | 需要合同、验收、付款、履约记录 |
| 监管线索 | 作为监管线索管理，不适合仅凭采购文件静态定性 |

## 状态
| 状态 | 含义 |
|---|---|
| 已设计 | 已纳入 B 总库，但尚未充分验证 |
| 候选 | 依据明确，可作为候选检查点 |
| 可启用 | 成熟度较高，可被业务系统优先启用 |
| 非静态 | 不适合仅通过采购文件静态审查，需要其他数据 |
| 已合并 | 被更高层级或更稳定检查点吸收 |
| 已废止 | 不再使用 |

## 成熟度
| 成熟度 | 判断标准 |
|---|---|
| 高 | 权威依据明确，审查问题清楚，常见表达较稳定，可用于业务系统优先启用 |
| 中 | 权威依据明确，但需要更多样本、上下文或边界说明 |
| 低 | 已有设计价值，但依据、样本、边界或数据条件仍不足 |

## 标准检查点
注册地、住所地、经营地限定供应商

## 检查点定义
本检查点用于审查采购文件是否以供应商注册地、住所地、经营地等地域条件限制或排斥供应商参与政府采购活动。

## 审查问题句
是否要求供应商注册地、住所地或经营地位于特定区域？

## 对应 A 边界
- [[rules/A类法源边界总表|A-BD-01 平等参与、公平竞争与禁止差别歧视边界]]

## 权威依据
- 财库〔2019〕38号
- 2025 年四类专项整治

## 审查对象
- 资格条件。
- 符合性审查。
- 评分因素。
- 实质性响应条款。
- 供应商承诺。

## 查找范围
- 采购公告。
- 供应商资格要求。
- 投标人须知。
- 资格审查表。
- 符合性审查表。
- 评分办法。
- 商务要求和服务要求。

## 定位关键词
- 注册地。
- 住所地。
- 经营地。
- 本地。
- 当地。
- 辖区内。
- 所在地。
- 驻地。

## 候选召回规则
出现地域类关键词，且该关键词与供应商主体条件、投标资格、得分、响应承诺、无效投标或履约准入直接关联时，应先召回为候选条款。

如果地域表述仅用于描述项目实施地点、交付地点、服务地点或采购人所在地，不直接作为供应商准入、评分或响应条件，不作为候选条款。

## 条款属性判断
判断候选条款属于以下哪类属性：
- 资格门槛。
- 评分因素。
- 实质性响应。
- 证明材料。
- 履约要求。
- 一般项目描述。

## 命中条件
满足以下条件之一，可以认定命中本检查点：
- 要求供应商注册地、住所地或经营地位于特定行政区域。
- 将本地注册、本地经营、本地驻地作为资格条件。
- 将本地注册、本地经营、本地驻地作为评分加分条件。
- 将不满足地域条件作为无效投标、不得分或负偏离。
- 以地域条件替代对真实履约能力的客观要求。

## 排除条件
出现以下情形时，不认定命中本检查点：
- 仅说明项目实施地点、服务地点、交付地点。
- 要求中标后在合理期限内建立服务响应机制，且未限定注册地、住所地或经营地。
- 法律法规对特定事项有明确地域管理要求。
- 地域表述只用于说明采购人地址、公告发布地区或合同履行地点。

## 正例
- “供应商须为本市注册企业。”
- “投标人须在本区设有经营场所，否则投标无效。”
- “注册地在本省的供应商得 3 分。”

## 反例
- “服务地点为采购人指定地点。”
- “中标人应在合同签订后建立本地服务响应机制。”
- “投标文件递交地点为本市公共资源交易中心。”

## 审查步骤
1. 检索地域类关键词。
2. 提取关键词所在条款及前后文。
3. 判断地域要求约束的是供应商主体，还是项目地点、服务地点、递交地点。
4. 判断该要求是否影响资格、评分、实质响应、无效投标或履约准入。
5. 检查文件是否提供了与项目履约直接相关的合理理由。
6. 对照命中条件和排除条件作出判断。
7. 如属于合理服务响应要求但边界不清，标记为待人工复核。

## 结论表达
命中时，应表述为“采购文件存在以供应商地域条件限制或影响供应商参与、评分或响应的风险”，并说明该地域条件出现的章节、条款属性和影响方式。不得脱离具体条款直接表述为违法结论。

## 审查模式
静态文件

## 成熟度
高

## 状态
可启用

## 适用边界
用于识别采购文件中以供应商注册地、住所地、经营地等地域条件限制供应商参与的情形。

## 备注
差别歧视高频检查点。

## 单页内容边界
B 单页应该包含：
- 标准检查点。
- 检查点定义。
- 审查问题句。
- 对应 A 边界。
- 权威依据。
- 审查对象。
- 查找范围。
- 定位关键词。
- 候选召回规则。
- 条款属性判断。
- 命中条件。
- 排除条件。
- 正例。
- 反例。
- 审查步骤。
- 结论表达。
- 审查模式。
- 成熟度。
- 状态。
- 适用边界。
- 备注。

B 单页不应该包含：
- 组合配置。
- 项目样本展开。
- 长篇案例分析。
- 模型执行过程。
- 历史实验页面来源。
- 旧名称或兼容关系。
- 未经重写的单项目风险结论。

## 维护规则
| 规则 | 说明 |
|---|---|
| 编号稳定 | B 编号一旦发布，不随名称调整而改变 |
| 名称可修订 | 标准检查点名称可优化，但需保留编号 |
| 状态可更新 | 根据依据、样本、业务反馈调整状态 |
| A 边界必须存在 | 没有 A 边界的检查点不得进入 B 总库 |
| 权威依据必须可追溯 | 权威依据优先回到 law-wiki source |
| 非静态要标明 | 需要投标、交易或履约数据的检查点不得标为静态文件 |
| 总库和单页同步 | 总库字段与单页 frontmatter / 正文保持一致 |

## 业务系统使用方式
业务系统可以：
- 读取 `B层标准检查点总库.md` 获取全量检查点。
- 按 `BD编号` 读取单个 `checkpoints/BDxx-yyy.md`。
- 按 `BD 问题域` 分组展示。
- 按 `状态` 选择启用范围。
- 按 `审查模式` 决定是否适用于当前任务。
- 按 `对应A边界` 展示权威归属。

## 结论
B 产品线保持纯粹：

```text
B 是标准检查点库。
B 回答具体查什么、怎么查、怎么判。
B 每一项都必须有 A 边界和权威依据。
B 可以被业务系统自由组合。
```
